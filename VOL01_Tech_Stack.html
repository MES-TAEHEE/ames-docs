VOL.1 · A-MES Architecture & Technology Stack
TECHNOLOGY STACK
Implementation stack for the A-MES three-system architecture — Office Web, Shop-Floor POP, and Logistics PDA. All components target the .NET 8/9 runtime and Microsoft SQL Server. Development is standardized on Visual Studio 2022 with optional Rider/VS Code support. The implementation team is composed of .NET / MSSQL specialists; tooling choices favor long-term familiarity, native debugging, and minimal external-runtime dependencies.

Why three separate clients? Office workers need a rich data-grid UI on large monitors (Web). Operators need large touch targets on fixed terminals (POP) without browser chrome. Logistics workers need a handheld with barcode scanner integration and offline tolerance (PDA). A single UI cannot serve all three well, so we share the back-end (API + DB + Identity) and tailor each front-end to its environment.
▼ SYSTEM 1 · OFFICE WEB
🖥️
SYS.1 · OFFICE WEB · 74 SCREENS
Blazor Server on .NET 9
PP · MNT · RPT · SYS · MD modules · Desktop browser (1920px)
LayerSelectionRationale
Runtime.NET 9 (LTS-class)Latest stable runtime, free upgrade path from .NET 8 LTS. Compatible with WinForms POP and MAUI PDA — single SDK across stack.
FrameworkBlazor ServerC# end-to-end (no JavaScript framework). Component model resembles WinForms — knowledge transfer to POP team. SignalR built in for live dashboards (PP-OEE, PP-ODM, RPT-07, MNT-09). Hot Reload + IntelliSense + native VS debugger.
UI LibraryMudBlazor 6.x (free, MIT)Material-design components. Includes MudDataGrid (sort/filter/paging out of the box for 14+ table-heavy screens), MudChart (Pareto/OEE), MudDatePicker, theming. Alternative: Syncfusion Blazor (commercial, richer Gantt/scheduler if PP-LSB or MNT-05 PM calendar needs it).
Real-timeSignalR (Blazor Server intrinsic)Every Blazor Server circuit already uses SignalR — pushing live updates to live dashboards (OEE/Andon/alarms) requires zero extra plumbing.
Auth UIASP.NET Identity + Razor pagesLogin/logout/password reset via scaffolded Identity UI. Custom branding overlay matching A-MES.
ChartsMudBlazor Chart, ChartJs.Blazor (fallback)MudBlazor's built-in chart handles 80% of cases (line/bar/pie/donut). For complex Pareto or stacked OEE, ChartJs.Blazor wraps the de-facto JS chart library while keeping the binding in C#.
Reporting (PDF)QuestPDF (free for < $1M ARR) or Telerik ReportingFluent C# API for generating delivery notes, BOL, daily production reports. Embeds smoothly in Blazor — no JS required.
Excel I/OClosedXMLFor RPT export / SYS-05 audit export / MD bulk import. Free, MIT.
Background jobsHangfireFor PP-02 SAP nightly imports, RPT scheduled emails (SYS-06 Notification), daily snapshots. Dashboard UI included.
LoggingSerilog + Seq (optional)Structured logging. Audit Log (SYS-05) writes to MSSQL via dedicated SP. Application logs to console + rolling file.
HostingKestrel behind IIS on Windows Server 2022IIS is the standard for on-prem .NET deployments in Korean enterprise. Windows Authentication available if cross-checking with AD becomes a future requirement.
Blazor Server tradeoff to plan for: each connected user holds an open WebSocket (SignalR circuit) to the server. For ~200 concurrent office users this is well within a single server's capacity (≈100MB RAM ceiling for sessions). If concurrent users approach 1,000+, add a load balancer with sticky sessions or move to Blazor WebAssembly + Web API later.
▼ SYSTEM 2 · SHOP-FLOOR POP
📟
SYS.2 · SHOP-FLOOR POP · 33 SCREENS
WinForms on .NET 8
INJ · IMG · PNT · QC processes · Fixed touch terminals on the line
LayerSelectionRationale
Runtime.NET 8 (LTS until Nov 2026)LTS recommended for long-lived fixed terminals. Designer support in Visual Studio is best in .NET 8.
FrameworkWinForms (.NET 8)Modern WinForms preserves the classic designer experience but runs on cross-cutting modern .NET. Drag-and-drop UI design matches what the team already knows.
Touch UICustom large-button styling + HighDpiMode.PerMonitorV2Buttons ≥ 64px tall, font ≥ 16pt, dark theme (matches POP demo). Avoid system dialogs — use full-screen kiosk shell.
API clientHttpClient + System.Text.JsonAll data goes through the shared Web API — POP never talks to MSSQL directly. Polly retry policy on transient network errors.
AuthJWT bearer (operator employee ID + 4-digit PIN, or RFID badge)Short-lived JWT (8 hr shift). RFID badge tap triggers /auth/badge endpoint. PIN keypad on screen for password.
BarcodeHID keyboard-wedge scanners (default)Scanned barcode appears as typed text in focused input. No SDK needed. For 2D imager features, Honeywell/Zebra SDKs available as optional add-on.
Charts (live)LiveCharts2 (free, MIT)For INJ-07 Production Status mini-charts on the POP. Smooth WinForms integration via CartesianChart control.
Andon (INJ-08)SignalR client (Microsoft.AspNetCore.SignalR.Client)POP connects to the same SignalR hub the Office Web uses. Andon call is pushed to all subscribed dashboards within ≤ 200 ms.
Offline cache (light)Local JSON file (per workstation)Master data (work-order list, mold list, defect codes) cached locally so a brief network blip doesn't block production entry. Posts queue and retry on reconnect.
LoggingSerilog + rolling file (local) + Sink.Http to APILocal file survives WAN outage; HTTP sink ships logs centrally when online.
DeploymentClickOnce from internal IIS shareEach POP terminal auto-updates from a single network publish URL. No per-terminal manual install. Roll back by re-publishing previous version.

▼ SHARED BACKEND · API · DATA · AUTH
🧩
SHARED · BACKEND API
ASP.NET Core Web API
One project, three clients — single source of business logic
LayerSelectionRationale
Runtime / FXASP.NET Core 9 (Web API)Hosted under Kestrel + IIS reverse proxy. Same Visual Studio solution as Office Web project for development convenience.
API styleController-based REST + selective Minimal API endpointsControllers for resource CRUD (familiar pattern). Minimal API for high-frequency utility endpoints (e.g., scan validation).
Data accessStored Procedures + ADO.NET (SqlConnection / SqlCommand)Per team decision: business rules live in stored procedures (DBA-managed, transactionally explicit). C# layer is thin SP invoker. Use Dapper as a lightweight mapper if the SP call/result mapping becomes repetitive (still SP-driven — Dapper just maps the result rows).
DB schema mgmtSSDT (SQL Server Database Project) + DACPACSchema is checked into the Git repo as a .sqlproj. Deploy compares the DACPAC to the target DB and generates upgrade script. Code-review friendly. No EF migrations needed (we're SP-driven).
IdentityASP.NET IdentityUser accounts stored in AspNetUsers/AspNetRoles tables in the same MSSQL. Password hashed (PBKDF2). Role-based authorization via [Authorize(Roles=...)] attributes. SYS-01 (User Mgmt) and SYS-02 (RBAC) screens manipulate these tables.
Token issuanceJWT bearer for POP / PDA / external API clientsWeb API exposes /auth/login that returns a JWT for non-browser clients. Office Web uses Identity cookies natively. Token includes role claims and a short expiry (8 hr).
ValidationFluentValidationDeclarative request validation, decoupled from the controller. Easier code review than DataAnnotations attributes scattered in DTOs.
Real-timeSignalR Hub (1 hub, multiple groups)Single hub /hubs/realtime. Groups: oee.line-{id}, andon.plant-{id}, alarm.{user}. Office Web, POP, PDA all subscribe.
Background jobsHangfire (MSSQL storage)SAP nightly import (PP-02), report subscriptions (SYS-06), PM-due notifications (MNT-05), daily snapshots. Web dashboard for monitoring jobs.
LoggingSerilog → file + MSSQL + Seq (optional)Structured logs (JSON). SYS-05 Audit Log writes to a separate SP_AUDIT_INSERT for compliance — kept distinct from app logs.
OpenAPISwashbuckle (Swagger UI at /swagger)Auto-generated API reference for developers. Disable in production or restrict by role.
▼ AUTHENTICATION & AUTHORIZATION FLOW
1
Web user login · Office Web user types ID/password in scaffolded Identity Razor Page. Identity validates against AspNetUsers (PBKDF2 hash). On success, sets .AspNetCore.Identity.Application cookie. Cookie carries role claims via ClaimsPrincipal.
2
POP / PDA login · Operator scans employee ID and enters 4-digit PIN (or taps RFID badge). Client POSTs to /api/auth/login with credentials → API verifies via Identity's SignInManager.CheckPasswordSignInAsync → returns a JWT with sub, role, shift, line claims and 8-hour exp.
3
Per-request authorization · Each API endpoint declares [Authorize(Roles = "Operator,Supervisor")] or [Authorize(Policy = "QC.Approve")]. Custom policies map to RBAC permission matrix maintained in SYS-02. Identity middleware enforces before controller runs.
4
SignalR auth · WebSocket upgrade carries the bearer token via query string (?access_token=...) for POP/PDA, or the Identity cookie for Office Web. Hub OnConnectedAsync reads claims and adds the connection to the correct group (e.g., oee.line-INJ-01).
5
Audit logging · Every authenticated mutation invokes IAuditLogger.LogAsync(userId, action, target, ipAddress) middleware → writes to tbl_AuditLog via SP_AUDIT_INSERT. Visible in SYS-05.
6
Refresh / revoke · JWT does not auto-refresh — operators re-sign-in at shift change. To revoke before expiry (e.g., user disabled in SYS-01), token is added to a small in-memory blocklist checked by a custom ITokenValidator on each request.

▼ DEPLOYMENT TOPOLOGY
🖥️
OFFICE WEB
  • Host: Windows Server 2022 + IIS 10 reverse-proxy → Kestrel
  • SSL: Wildcard cert on IIS
  • URL: https://ames.seyonehwa.us
  • Scale: 1 server initially; sticky-session LB if > 500 concurrent
  • Update: Web Deploy package via Azure DevOps release pipeline
📟
SHOP-FLOOR POP
  • Host: Each line has a fixed Windows 11 IoT terminal (touch monitor)
  • Distribution: ClickOnce published to internal IIS share
  • URL: https://ames.seyonehwa.us/pop/install
  • Update: Terminal checks for update on launch; auto-installs new version
  • Rollback: Re-publish previous version
📱
LOGISTICS PDA
  • Device: Zebra TC52ax / Honeywell CT45 (Android 13)
  • Distribution: Signed APK pushed via MDM (SOTI / Workspace ONE / Zebra StageNow)
  • Update: MDM tag drives version rollout (canary → 10% → 100%)
  • Offline: Local SQLite cache; transactions queue and sync on Wi-Fi recovery
  • Crash reporting: AppCenter Diagnostics (optional)
CI/CD pipeline: All three projects build in a single Azure DevOps pipeline (or GitHub Actions). Triggers on push to main: (1) Web API + Web App → Web Deploy package → IIS, (2) WinForms POP → ClickOnce publish, (3) MAUI Android → signed APK uploaded to MDM endpoint. DACPAC for the database is applied to staging on PR merge, to production on tagged release.

▼ SOLUTION STRUCTURE
Recommended single Visual Studio solution layout — one .sln spans all three clients + shared API + DB project, enabling F5-to-debug across the stack. 
A-MES.sln
├── 01_Shared/
│   ├── AMES.Contracts/           # DTOs, enums, shared between API and clients
│   └── AMES.Common/              # JWT helpers, validation, audit interfaces
│
├── 02_Database/
│   └── AMES.Database/            # SSDT .sqlproj — schema, SPs, seed data → DACPAC
│
├── 03_Api/
│   └── AMES.Api/                 # ASP.NET Core Web API + SignalR hub + Hangfire
│
├── 04_OfficeWeb/
│   └── AMES.OfficeWeb/           # Blazor Server (74 screens) + ASP.NET Identity UI
│
├── 05_PopWinForms/
│   └── AMES.Pop/                 # WinForms (.NET 8) for POP terminals
│
├── 06_PdaMaui/
│   └── AMES.Pda/                 # .NET MAUI (Android) for handheld PDA
│
└── 07_Tests/
    ├── AMES.Api.Tests/           # xUnit + WebApplicationFactory
    ├── AMES.OfficeWeb.Tests/     # bUnit for Blazor components
    └── AMES.Integration.Tests/   # End-to-end (testcontainers MSSQL)
Why one solution? Allows F5 debugging across boundaries (e.g., set breakpoint in API, drive from Blazor Web). Single dotnet build validates all clients still compile against the shared contracts. Each project still publishes independently.
