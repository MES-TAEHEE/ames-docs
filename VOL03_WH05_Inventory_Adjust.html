01
Overview
화면 개요
Manual correction of system inventory to match physical count results. Every adjustment requires a reason code (one of: COUNT_DIFF, DAMAGE, EXPIRED, SYSTEM_ANOMALY, RELOCATION) and supervisor approval before commit. Adjustments above ₩500,000 in value or 10% of on-hand qty automatically escalate to plant manager approval. All adjustments are written to WH-08 transaction history with before/after values, photos (optional), and approver signature.
실사 결과를 시스템 재고에 반영하는 수동 조정 화면입니다. 모든 조정은 사유 코드(COUNT_DIFF 실사 차이 · DAMAGE 파손 · EXPIRED 유통기한 · SYSTEM_ANOMALY 시스템 이상 · RELOCATION 위치 이동)와 관리자 승인 후 확정됩니다. 50만원 초과 또는 현재고 10% 초과 조정은 공장장 승인으로 자동 에스컬레이션. 모든 조정은 전/후 값, 사진(선택), 승인자 서명과 함께 WH-08에 기록됩니다.

04
Event & Action Flow
이벤트 · 액션 흐름
1
Userclick · [+ New Adjustment]
Warehouse staff opens new adjustment form. Requires WH.Adjust role.
창고 담당자가 신규 조정 폼을 엽니다. WH.Adjust 권한 필요.
UI Empty form rendered · item code autocomplete focused.
2
Userselect · item + location
User selects item and location. System fetches current qty and unit cost. Form recalculates Δ and value impact live as counted qty is entered.
품목·위치 선택 시 현재고와 단가 자동 로드. 실사 수량 입력에 따라 Δ·금액 실시간 계산.
APIGET /api/wh/inventory?item=...&loc=....
3
Userselect · reason code
User selects reason. If DAMAGE/EXPIRED, photo upload becomes required. If SYSTEM_ANOMALY, incident ticket ID field appears.
사유 선택 시 DAMAGE/EXPIRED는 사진 필수, SYSTEM_ANOMALY는 인시던트 티켓 ID 입력 필드 표시.
UI Form fields conditionally enabled/required.
4
Userclick · [Submit for Approval]
All validations pass. System checks escalation thresholds: if |value| > ₩500K or |Δ/current| > 10%, routes to Plant Manager; else to Supervisor.
전체 검증 통과 후 에스컬레이션 임계 확인: 금액 50만원 초과 또는 비율 10% 초과 → 공장장, 그 외 → 슈퍼바이저로 라우팅.
APIPOST /api/wh/adjustment · status='Pending' · approver_level=1|2.
5
Systemnotify · approver
Approver receives in-app notification + email. Approval queue card appears on their WH-05 view. SLA: 4 business hours for Supervisor, 1 business day for Plant Mgr.
승인자에게 인앱·이메일 알림 + 승인 대기 카드 표시. SLA: 슈퍼바이저 4시간, 공장장 1영업일.
SIDE-EFFECT SLA timer starts · auto-reminder at 75% elapsed.
6
Approverclick · [✓ Approve]
Approver reviews detail + photos. Clicks Approve. System asks confirmation, then commits inventory change atomically in a transaction.
승인자가 상세·사진 확인 후 [승인] 클릭. 확인 다이얼로그 후 트랜잭션으로 재고 변경 원자적 커밋.
APIPOST /api/wh/adjustment/:id/approve · BEGIN TRAN; UPDATE wh_inventory; INSERT wh_transaction; COMMIT;
✓
Systemcomplete · downstream
Inventory updated. WH-08 transaction logged with reason code, approver signature, before/after values. Requester notified. Below-min check re-evaluated.
재고 갱신. WH-08에 사유·승인자 서명·전후 값 기록. 요청자에게 결과 알림. 최소재고 재평가.
DOWNSTREAM WH-03/WH-04 push update · WH-08 audit row · BR-WH-03 re-check.
05
Exception & Error Handling
예외 · 에러 처리
E-WH05-001
Counted Qty = Current실사·현재 동일
Cause
User enters counted qty equal to current. No adjustment needed. 실사 수량이 현재고와 동일.
Detect
Form validation on Submit.
Action
Block submit. Modal: "No adjustment needed — counted qty equals current. Cancel or correct?"
Recovery
User cancels or enters different qty.
⛔ NO Δ — CANCEL OR CORRECT
E-WH05-002
Negative Result Stock음수 재고 결과
Cause
Counted qty < 0, or system on_hand would become negative on commit. 결과 재고가 음수.
Detect
Server validation; DB CHECK constraint also enforces (BR-WH-05).
Action
Block. "Counted qty must be ≥ 0. Inventory cannot be negative."
Recovery
User re-counts physical stock.
⛔ NEGATIVE NOT ALLOWED
E-WH05-003
Photo Required (Missing)사진 누락
Cause
Reason is DAMAGE or EXPIRED but no photo attached. 파손·만료 사유인데 사진 미첨부.
Detect
Form validation on Submit.
Action
Block submit. Highlight photo field red. Inline error: "Photo evidence required for DAMAGE/EXPIRED."
Recovery
User attaches photo or changes reason.
⚠ PHOTO REQUIRED
E-WH05-004
Concurrent Modification동시 수정 충돌
Cause
Inventory changed (e.g., WH-02 commit) between form open and submit. 폼 열림과 제출 사이에 재고 변경.
Detect
Optimistic lock: version column mismatch.
Action
Reject submit. Modal: "Inventory changed. Current qty is now X. Re-count or adjust?" Show new value.
Recovery
User refreshes data and reconsiders.
⚠ STALE DATA — REFRESH
E-WH05-005
Approval SLA Breached승인 SLA 초과
Cause
Approval pending past SLA (4hr Supervisor / 1day Plant Mgr). SLA 시간 초과.
Detect
Hourly cron checks pending_since.
Action
Auto-escalate to next level (Supervisor → Plant Mgr, Plant Mgr → Plant Director). Email all parties.
Recovery
Next-level approver acts on request.
⚠ SLA BREACH — AUTO-ESCALATED
E-WH05-006
Self-Approval Attempt자체 승인 시도
Cause
Approver attempts to approve own adjustment request. 요청자가 자신의 요청을 승인 시도.
Detect
Server check: approver_id == requester_id.
Action
Block. "Self-approval is not allowed. Route to another approver." Audit logged.
Recovery
Different approver handles request.
⛔ SELF-APPROVAL FORBIDDEN

07
Applicable Business Rules
적용 업무 규칙
Rule IDDescriptionAction
BR-WH-05
Inventory quantity must never be negative. DB-level constraint rejects any adjustment that would result in negative on-hand.
재고는 음수 불가. DB 제약으로 음수 결과 조정 거부.
Block
BR-WH-10
All inventory adjustments require reason code + supervisor approval. Adjustments with |value| > ₩500K or > 10% of on-hand automatically escalate to plant manager.
모든 재고 조정은 사유코드 + 슈퍼바이저 승인 필수. 50만원 또는 10% 초과 시 공장장 자동 에스컬레이션.
Workflow
BR-WH-11
DAMAGE and EXPIRED reasons require photo evidence (min 1, max 3 photos). Photos retained 5 years for audit.
파손·만료 사유는 사진 증빙 필수(1~3장). 감사용 5년 보관.
Required
BR-WH-12
Self-approval is strictly forbidden. The requester cannot be the approver for the same adjustment.
자체 승인 금지. 요청자와 승인자가 동일인 불가.
Block
